# Architecture

## Overview

Job Tracker is a standard client-server application with a clear separation between the React SPA and the Express API. The database is the source of truth; the API layer owns all business logic; the client is a thin presentation layer.

## Core Principles

**1. Server owns logic.** No business logic in the client. The API validates, enforces rules, and is the single point of access to the DB.

**2. Migrations over manual schema.** Every schema change is a numbered migration file. The DB state is always reproducible from scratch.

**3. Status history is append-only.** When a job moves from "Applied" to "Phone Screen," a row is inserted into `status_events` вҖ” never updated. This gives a full audit trail and enables timeline views.

**4. AI is a feature, not a dependency.** The AI layer is a single optional service. If `CLAUDE_API_KEY` is not set, that route returns 501. The app is fully functional without it.

**5. Auth is stateless.** JWTs issued at login, verified by middleware on protected routes. No sessions, no cookies. Short-lived access tokens + refresh token rotation.

## Observability

The app uses **OpenTelemetry (OTel)** as its observability foundation вҖ” traces are the primary signal, not logs. Logs and metrics are secondary and derived where possible from trace data.

### Trace-First Philosophy

Every meaningful operation is wrapped in a span. Spans are stitched into traces via a **correlation ID** (`x-correlation-id`) that flows from the client through every layer:

```
Browser вҶ’ [x-correlation-id: abc-123]
  вҶ’ Express middleware (reads or generates correlation ID)
    вҶ’ route handler span
      вҶ’ service layer span
        вҶ’ DB query span
          вҶ’ (optional) Claude API span
```

The correlation ID is:
- Generated by the client on each request (UUID v4) if not already present
- Accepted from the client via `x-correlation-id` request header
- Propagated through all internal spans as a trace attribute
- Returned in the response as `x-correlation-id` so the client can log it
- Attached to any client-side error reports (see below)

This means a single correlation ID lets you reconstruct the full picture of any request вҖ” client error, API call, DB query, and external API call вҖ” in one query in your trace backend.

### Server-Side Tracing

- **`@opentelemetry/sdk-node`** auto-instruments Express, `pg`, and outbound HTTP
- Spans exported to **[Grafana Tempo](https://grafana.com/oss/tempo/)** (free, OSS) or **Jaeger** locally via OTLP
- In production: export to [Grafana Cloud](https://grafana.com/products/cloud/) free tier (50GB traces/month) or [Honeycomb](https://www.honeycomb.io/) free tier
- Key custom attributes on all spans:
  - `correlation_id` вҖ” ties client вҶ” server вҶ” DB
  - `user_id` вҖ” scoped to authenticated requests (never email or PII)
  - `http.route` вҖ” normalized route pattern (not raw URL, avoids ID leakage into span names)

### Client-Side Error Reporting

Client-side exceptions are invisible to the server by default. We address this with a lightweight error reporting pipeline:

**Mechanism:**
1. React app wraps the tree in an `ErrorBoundary` component
2. `window.onerror` and `window.onunhandledrejection` catch anything outside React's lifecycle
3. On error, the client POSTs to **`POST /api/errors`** (unauthenticated, rate-limited) with:
   - `message` вҖ” exception message
   - `stack` вҖ” stack trace (source-mapped in production вҖ” see below)
   - `correlationId` вҖ” the last known correlation ID, so the error can be tied to the triggering request
   - `route` вҖ” current browser route (e.g. `/jobs/abc-123`)
   - `userAgent` вҖ” browser/OS context
4. The server receives the report, attaches it as a span event, and records it as a trace вҖ” **not** a raw log

**Source maps:** Vite generates source maps at build time. In production, source maps are uploaded to the trace backend (or a private storage bucket) and **not** served publicly вҖ” the browser sends the minified stack, the backend resolves it server-side. This avoids exposing source code via public `.map` files.

**What's never sent in error reports:**
- JWT tokens
- Password fields
- Resume text
- Any input field contents

**Rate limiting on `/api/errors`:** 10 requests per IP per minute. Errors are deduplicated by message + stack hash before recording to avoid noise storms from repeated UI failures.

### What Never Appears in Traces or Logs

- Passwords or password hashes
- JWT tokens (access or refresh)
- Request/response bodies on auth routes
- Resume text sent to the AI endpoint
- Any PII beyond `user_id` (opaque UUID) on authenticated spans

## Security

See [SECURITY.md](SECURITY.md) for the full OWASP Top 10 analysis.

Summary: the primary surface area is auth (JWT handling, password storage) and access control (ensuring users can only access their own data). Injection is mitigated by parameterized queries throughout. The app has no file uploads, no XML parsing, and no third-party component complexity beyond its direct dependencies.

## Build Phases

### Phase 1 вҖ” Core CRUD (no auth)
Get the data model right. Build the jobs table, status_events, and basic CRUD routes. Seed with test data. Validate the schema makes sense before writing any UI.

### Phase 2 вҖ” Auth
Add users table. Protect all routes. Implement login/register with bcrypt + JWT.

### Phase 3 вҖ” React Frontend
Build the UI against the running API. Job list, detail view, add/edit form, status history timeline.

### Phase 4 вҖ” AI Layer
Single POST endpoint: accepts a job ID + resume text, returns a cover letter. On-demand only.

### Phase 5 вҖ” Deployment
Railway (Postgres + Node service). Environment variables for secrets. Production build pipeline.
